## 面试里一提消息重复，很多人第一反应就是加锁

问到“消费者如何保证幂等”时，最常见的回答是上分布式锁。这个答案不能说完全错，
但如果把它当成默认起点，通常说明对问题的本质还没抓稳。幂等处理的核心不是“同一时刻只能有一个线程执行”，
而是“同一条业务结果无论被处理多少次，最终只落一次正确状态”。

换句话说，锁解决的是并发竞争，幂等解决的是重复执行后的结果一致。两者有关联，但不是一回事。
很多线上重复消费问题，根因并不是多个线程同时抢着改一条数据，而是消息重投、消费超时重试、
生产端重复发送，或者消费成功但 ack 丢失。此时如果只想着先上锁，很容易把简单问题做重，
却没有建立真正可靠的落库边界。

## 先找业务唯一键，才能知道“重复”到底怎么判

幂等的第一步，通常不是写锁代码，而是先明确业务唯一键。比如订单支付结果消息，
唯一键可能是 `orderId + paymentStatus`；库存扣减回执，唯一键可能是 `bizOrderNo`；
发券消息则可能是 `userId + couponBatchId + bizTraceNo`。

如果连“哪几个字段共同定义一次业务动作”都说不清，后面的锁、去重表、状态机都容易做偏。
因为系统里的消息 ID 往往只代表“这条消息对象”，不一定代表“这次业务行为”。有些重试消息虽然消息 ID 不同，
但业务含义完全相同；也有些同一个订单会对应多种状态推进，不能简单拿订单号一把拦死。

面试里更稳妥的表述是：我会先识别业务唯一键，再围绕这个键设计幂等落点，而不是直接把消息 ID 当唯一依据。

## 为什么通常先查业务键

先查业务键，本质上是在判断“这件事是不是已经做完了”。这个判断一旦能成立，很多重复消息根本不需要进入复杂流程。

最典型的做法有两类：

1. 查询业务表状态。比如订单已经是“已支付”，那重复的支付成功消息就可以直接返回成功。
2. 查询独立幂等表。比如以 `biz_key` 建唯一索引，已经存在成功处理记录就直接跳过。

这样做的好处是简单、可解释，而且和业务结果直接关联。你不是靠“我猜现在没人和我并发”，
而是靠“数据库里已经有了这次业务的最终事实”来判断是否重复。

很多系统里，真正稳定的幂等能力就来自数据库唯一约束和业务状态检查，而不是复杂的分布式锁编排。

## 只查不够，关键是把“查”和“写结果”放进同一边界

当然，只做一次查询并不自动安全。因为两个消费者可能几乎同时查到“还没处理”，然后一起往下执行。
所以更准确的说法不是“查完就行”，而是“用业务键判断重复，并把最终落库放进可约束的原子边界里”。

常见落法有三种。

第一种，业务表天然就有唯一约束。比如一条转账流水只能插一次，
那就直接依赖唯一索引插入，插入成功代表首次处理，插入冲突代表重复消息。

第二种，先写幂等记录表，再执行业务逻辑，但要处理好“记录写成功、业务执行失败”的补偿问题。
这适合需要清晰追踪消费轨迹的场景。

第三种，把状态更新设计成条件更新。比如只有订单状态仍为“待支付”时，才能更新成“已支付”，
更新影响行数为 1 说明本次成功，等于 0 说明别人已经处理过或状态已推进。

这三种方式有一个共同点：它们都让“是否首次处理”最终由数据库约束或状态机来裁定，
而不是只靠应用层 if 判断。

## 那什么时候才需要锁

锁不是不能用，而是应该在数据库唯一约束和状态边界不足以覆盖风险时再补上。

比如下面几种情况，锁会更有意义：

1. 业务逻辑里有长链路调用，且重复执行代价很高，例如会重复调用外部资金接口。
2. 最终结果不是一次数据库写入能收住的，中间包含多个不可逆步骤。
3. 下游系统没有天然幂等能力，而你又必须尽量减少重复触发。

即便如此，锁通常也只是保护窗口，不应该替代最终落库约束。因为锁会过期，会丢失，会因为网络抖动产生误判，
而数据库里的唯一索引和业务状态才是最后的裁判。

如果面试官继续追问，我会补一句：锁可以减少并发重复执行，但真正闭环幂等的，还是业务唯一键加持久化约束。

## 一个常见误区：把“去重成功”当成“业务成功”

实际系统里还有一个很容易踩的坑：幂等记录写入成功，不等于业务已经成功。

例如你先插入一条 `consume_record` 表，标记某个 `biz_key` 正在处理，然后开始扣库存、写订单、发通知。
如果中途失败，而你又没有设计状态机或重试恢复，这条记录反而会把后续重试全部挡掉，
导致消息看起来“被去重了”，实际业务却没做完。

所以幂等设计通常至少要区分：

1. 处理中
2. 处理成功
3. 处理失败可重试

面试里如果能把这层说出来，基本就比“加个锁就好了”要深入得多。因为你已经意识到，
幂等不是一个布尔开关，而是一套围绕业务结果收敛的状态控制。

## 更完整的回答方式

如果被问“消费者幂等处理为什么先查业务键而不是先上锁”，可以这样回答：

我会先识别这条消息对应的业务唯一键，再用业务表状态、唯一索引或幂等记录判断它是否已经处理过。
原因是幂等关注的是最终业务结果是否重复落地，而不是单纯限制并发。锁只能减少并发窗口内的重复执行，
不能替代持久化约束。真正稳妥的做法，是把基于业务键的判断与最终写入放进同一原子边界，
必要时再用分布式锁降低外部调用或长流程中的并发风险。

这个回答的重点，不是否认锁，而是把锁放回它该在的位置。
