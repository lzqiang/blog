## 很多人踩坑都从 `0.0` 和 `0.00` 开始

Java 里一说到精确计算，大家很自然会想到 `BigDecimal`。这一步没有问题，问题出在很多人把它当成
“带小数点的整数”来使用，尤其是在比较大小或判断相等时，顺手就写出 `a.equals(b)`。

表面上看，这很合理：两个金额都是零，为什么不相等？可 `BigDecimal` 的 `equals` 比较的不只是数值，
还比较 `scale`。也就是说，`new BigDecimal("0.0")` 和 `new BigDecimal("0.00")` 在数学意义上相等，
在 `equals` 看来却不是同一个值。

这类问题之所以隐蔽，是因为大多数业务在打印日志、展示页面、调试断点时都只会看到“0”。只有到了集合去重、
缓存命中、条件分支或者测试断言时，差异才会突然冒出来。

## `equals` 到底在比什么

`BigDecimal` 内部既保存数值，也保存精度信息。`1.0` 和 `1.00` 虽然表示相同的数学结果，
但它们携带的小数位不同。`equals` 设计成严格对象相等，因此它会同时比较：

1. 非缩放数值是否一致。
2. 小数位数，也就是 `scale` 是否一致。

所以：

```java
new BigDecimal("1.0").equals(new BigDecimal("1.00"))   // false
new BigDecimal("1.0").compareTo(new BigDecimal("1.00")) // 0
```

`compareTo` 才是“数值意义上的比较”。返回 `0` 代表数值相等，小于 `0` 代表左边更小，大于 `0` 代表左边更大。

## 线上最常见的误用场景

第一个常见场景是金额判断。比如订单退款后要检查“剩余待退金额是否为 0”，有人会这样写：

```java
if (remainingAmount.equals(BigDecimal.ZERO)) {
    // 标记已结清
}
```

如果 `remainingAmount` 是通过数据库字段、接口传参或计算链路得到的 `0.00`，这段代码就会失效。
页面上金额已经归零，状态却迟迟不切换，排查起来还很容易误以为是事务或缓存问题。

第二个场景是阈值判断。比如某个折扣门槛要求“满 100 元”，你把金额做标准化时保留了两位小数，
但配置里的阈值是 `100`，随后用 `equals` 判断是否“恰好命中”。业务上以为比较的是金额值，
实际上比较的是金额值加表示方式。

第三个场景是集合行为。把 `BigDecimal` 放进 `HashSet`、作为 `Map` 的 key，或者拿来做对象去重时，
`equals` 和 `hashCode` 是配套的。`10.0` 和 `10.00` 会被当成两个不同元素，这在账务类、报表类系统里尤其危险，
因为你眼里的“一笔金额”可能在程序眼里变成了两份记录。

## 为什么有些人明明知道这个坑，代码里还是会出现

根因通常不是“不知道 `compareTo`”，而是对象相等和业务相等被混在了一起。写业务代码时，人会本能地把
“两个值一样”理解成“应该用 `equals`”。可 `BigDecimal` 恰好是一个反例：对象语义很严格，业务语义却往往只关心数值。

另外，很多团队有一个习惯是把 `BigDecimal` 当普通值对象塞进 DTO、VO、实体和缓存模型里，却没有约定统一的创建方式。
有人用 `"0.00"`，有人用数据库反序列化结果，有人从 `BigDecimal.ZERO.setScale(2)` 得到值。来源一多，
`scale` 就开始漂移，最后在某个不起眼的分支里炸出来。

## 更稳妥的写法是什么

如果你要比较数值大小，或者判断数值是否相等，优先使用 `compareTo`：

```java
if (remainingAmount.compareTo(BigDecimal.ZERO) == 0) {
    // 数值上等于 0
}

if (amount.compareTo(limitAmount) >= 0) {
    // 大于等于门槛
}
```

如果业务要求的是“表示形式也要一致”，比如导出模板校验、财务报文格式核对，这时才应该用 `equals`，
因为你要比较的已经不只是值，而是值的表达方式。

## 只改比较方式还不够，还要管输入规范

很多 bug 不是出在最后那句 `equals`，而是出在整个系统没有统一金额规范。更稳的做法一般包括：

1. 创建 `BigDecimal` 时坚持使用字符串构造，避免 `double` 带来的二进制误差。
2. 在进入核心业务前统一 `scale` 和舍入规则，比如金额一律保留两位。
3. 在公共工具层封装金额判断方法，减少各处手写比较。
4. 对集合 key、去重键、缓存键明确约束，避免直接拿 `BigDecimal` 原样参与判等。

这些约束的价值在于，把“该比数值还是该比格式”这件事前置，而不是留给每个业务开发临场决定。

## 面试或 code review 里可以怎么回答

如果有人问“为什么 `BigDecimal` 比较不要直接用 `equals`”，比较完整的回答可以是：

`BigDecimal` 的 `equals` 会同时比较数值和 `scale`，因此 `1.0` 和 `1.00` 会被判定为不相等。大多数业务场景只关心数值，
所以应该使用 `compareTo`。如果直接用 `equals`，容易在金额归零判断、阈值判断、集合去重和缓存命中上出现隐蔽 bug。
只有在确实需要比较格式和精度表达时，`equals` 才是合适选择。

把这件事讲清楚，本质上不是记住一个 API 细节，而是分清“对象相等”和“业务相等”到底是不是一回事。
