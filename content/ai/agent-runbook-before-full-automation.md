## 许多失败的自动化，不是模型不够强，而是运行手册为空

团队第一次把 Agent 接进工作流时，注意力常常都放在提示词和模型能力上。大家会讨论应该选哪个模型、
上下文窗口够不够大、工具调用顺不顺畅，却很少先问一个更基础的问题：这个 Agent 在真实流程里，
到底被允许做到哪一步，失败以后又该怎么收场。

结果就是，试运行时看起来很聪明，真到持续运行阶段却问题不断。它可能把不完整的信息也当成可执行输入，
可能在权限边界附近来回试探，也可能遇到异常后反复重试，把一个局部错误扩大成整条链路的阻塞。

很多时候，缺的不是更复杂的 Prompt，而是一份面向运行阶段的手册。

## 运行手册首先要定义“它接什么输入”

对于 Agent 来说，输入不是一句自然语言描述那么简单。真正进入生产流程后，它拿到的往往是混杂的任务文本、
历史状态、外部工具返回值、权限上下文和部分结构化参数。如果这些输入质量不稳定，Agent 的输出也很难稳定。

所以运行手册的第一部分，应该明确规定：

1. 哪些字段是必填的。
2. 哪些上下文缺失时必须中止，而不是猜测补全。
3. 哪些外部工具结果需要做格式校验后才能继续。
4. 哪些历史状态只作参考，不能直接视为最新事实。

这一步的价值在于把“模型推理问题”前移成“输入治理问题”。很多后续异常，本质上都能在入口被拦住。

## 第二部分要写清楚它的权限边界

Agent 最危险的时刻，不是完全不会做事，而是“能做一点点事，但边界又没有被写死”。
例如它可以改文档、发评论、合并代码、执行脚本，表面上每项能力都合理，可一旦没有明确边界，
它就可能在低置信度场景下做出高影响动作。

因此运行手册不能只写“Agent 可以调用哪些工具”，还要写：

1. 哪些动作允许自动执行。
2. 哪些动作必须人工确认。
3. 哪些动作在任何情况下都禁止自动触发。
4. 发生连续失败或结果不一致时，权限是否应自动降级。

只有把这些规则写成能落地的约束，自动化才不会在压力下滑向“先做了再说”。

## 第三部分要提前设计失败路径，而不是事后补救

很多自动化上线前只验证成功流程，仿佛 Agent 的主要工作就是一路正确地跑到底。
但真实环境里，失败路径才是决定系统是否可靠的关键部分。

运行手册至少应回答下面几个问题：

1. 工具超时后，是重试、回退还是终止。
2. 多次得到相互矛盾的结果时，以哪个来源为准。
3. 外部系统部分成功时，如何避免重复副作用。
4. 达到重试上限后，如何把现场信息交给人工处理。

这套规则的意义，不只是减少故障，更是把故障变得可恢复。一个会失败但能干净交接的 Agent，
往往比一个平时很顺、出错时却没人知道它做过什么的 Agent 更值得信任。

## 人工接手方式要和自动化能力一起设计

很多团队把“人工兜底”理解成一句空话，觉得出了问题再让人接进来就行。可如果没有提前设计交接格式，
人工接手通常会非常痛苦。接手的人既不知道 Agent 已经做到了哪一步，也不知道哪些结论可信，
最后只能重新从头梳理。

更可行的做法是让运行手册要求 Agent 在中止时输出最小但完整的交接包，例如：

1. 当前任务目标和已确认约束。
2. 已调用过的工具及结果摘要。
3. 失败发生的具体步骤和最近一次错误。
4. 它建议的人类下一步检查点。

这样人工不是接管一团黑箱状态，而是接管一个已经整理过上下文的现场。

## 手册不是文档装饰，而是评估基线

一旦运行手册写清楚，它还会自然变成评估集的来源。你可以据此检查：

1. 输入缺字段时，Agent 是否会正确拒绝继续。
2. 高风险动作前，是否真的触发确认。
3. 工具失败时，是否按预期退避或交接。
4. 交接摘要是否足够让新人快速接手。

这比只统计“任务完成率”更接近真实可靠性。因为生产环境要的不是偶尔惊艳，而是长期可控。

## 自动化真正需要的，是可运行的秩序

在全自动之前，先把 Agent 的运行手册写出来，听上去像是在减慢推进速度，
其实是在避免后面用更多时间偿还混乱成本。模型能力决定上限，运行手册决定下限。

当输入要求、权限边界、失败路径和人工接手方式都被提前说清楚时，Agent 才不只是一个“会回答问题的模型”，
而是一个能稳定嵌入真实流程的系统部件。对多数团队来说，这一步往往比再堆一层提示词更重要。
