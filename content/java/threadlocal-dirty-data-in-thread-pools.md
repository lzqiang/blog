## 问题不在 ThreadLocal，本质在线程会被复用

很多人第一次接触 ThreadLocal，觉得它像是“线程私有变量”的快捷写法，于是顺手就把用户信息、
链路标识、租户编号甚至数据库路由键都塞进去。单看 API，这么做似乎很自然：当前线程 set，一路调用 get，
最后业务结束，好像事情就结束了。

真正的风险出现在你把它放进线程池环境以后。Web 容器、异步执行器、消息消费线程，绝大多数都不是
“新建线程，执行一次，再销毁”。它们更常见的模式是：少量工作线程反复处理大量任务。线程不死，
ThreadLocal 里那份数据就有机会跟着活得比请求更久。

## 为什么会出现“串请求”的脏数据

假设一个接口在入口处把 `userId` 放进 ThreadLocal，后面日志、鉴权和审计逻辑都从这里拿值。
如果这次请求处理结束后没有清理，线程回到池里等待下一个任务。等下一次这个线程再次被拿出来执行时，
新的代码如果先读后写，或者某个分支漏了 set，就可能直接读到上一次请求留下来的值。

这类问题最麻烦的地方在于，它通常不是稳定复现的。只有当两个请求刚好复用到同一条线程，并且后一个请求
又刚好走到那个漏设值的分支时，问题才会暴露出来。线上看起来就像“偶发拿错用户”“日志 traceId 串了”
或者“多租户查询命中了别人的库”，很难第一时间把锅定位到 ThreadLocal。

## 内存泄漏也常和它绑在一起被讨论

除了脏数据，ThreadLocal 还经常和内存泄漏一起出现。这里需要把话说准确一点：不是说用了 ThreadLocal
就必然泄漏，而是在线程生命周期很长时，ThreadLocalMap 里的 value 如果没有及时移除，就会跟着线程一起常驻。

尤其当你放进去的是较重对象，比如用户权限树、大对象缓存、数据库连接上下文或者自定义缓冲区时，
问题会更明显。即便 key 已经因为弱引用失效，value 仍可能在后续访问和清理前挂在线程上。
对于常驻线程池，这种“迟迟不释放”的效果在线上就足够糟糕了。

## 最常见的误区是“我在 finally 里偶尔会清理”

工程里真正危险的，不是完全没人知道要 `remove()`，而是大家只在一部分路径上记得做。比如：

1. 正常返回时清理了，但异常分支直接抛出。
2. Controller 层清理了，但中途又在异步线程里重新 set 了一份。
3. 一个过滤器写入，另一个拦截器假设别人会清理。
4. 框架升级后调用链顺序变化，原本的清理点不再覆盖所有出口。

这也是为什么 ThreadLocal 问题很少靠“提醒大家小心”来解决。只要清理责任是分散的，
迟早会有人漏掉一个分支。

## 更稳妥的做法是把 set 和 remove 放在同一层管理

如果你必须使用 ThreadLocal，最好把写入和清理封装成一对固定动作，放在请求入口和出口附近统一处理。
例如 Web 场景里由过滤器或拦截器成对管理，消息消费场景里由消费模板包住，异步任务场景里由任务装饰器控制。

关键点不是“有一个地方调用了 remove”，而是“这个 remove 必然和对应的 set 配对出现”。
只有当上下文生命周期和业务请求生命周期对齐，ThreadLocal 才不会把线程池复用的副作用传给下一个任务。

## 异步和线程切换会让问题再升级一层

还有一种常见误判是：主线程 set 了值，异步线程理所当然也能读到。普通 ThreadLocal 做不到线程间传播，
于是有人会改用 InheritableThreadLocal，甚至自己做上下文复制。这样确实能“传过去”，
但如果复制和清理策略没设计好，污染范围反而更大。

在线程池里，子线程并不是每次都新建的，继承关系并不等于请求边界。你以为是在传递上下文，
实际可能是在扩大旧数据残留的时间和范围。很多链路追踪、MDC 或安全上下文方案之所以要提供专门的包装器，
本质上就是为了在传播的同时显式回收。

## 可以把它当成“受控的全局变量”

我更喜欢把 ThreadLocal 理解成一种受控但危险的全局变量。它不是参数传递的替代品，只适合放那些
确实需要贯穿一段调用链、又不值得层层透传的上下文信息，而且这段调用链的起点和终点必须足够清晰。

一旦边界不清楚，或者线程模型比你想象得更复杂，它的收益就会迅速被副作用抵消。把业务核心数据
过度依赖 ThreadLocal，会让代码表面上更简洁，排查问题时却更隐蔽。

## 实战里可以记住这几个判断

如果线上怀疑和 ThreadLocal 有关，我通常先看四件事：

1. 这份上下文是谁 set 的，在哪些出口一定会 remove。
2. 执行载体是不是线程池，线程是否会长期复用。
3. 是否存在异步、回调、重试或跨线程切换。
4. 放进去的值是轻量标识，还是会明显占内存的大对象。

这四个问题一过，很多“偶发诡异问题”就会突然变得具体。

## 最后给一个简单结论

ThreadLocal 容易留下脏数据，不是因为它设计错了，而是因为它把数据生命周期绑定到了线程，
而线程池又把线程生命周期拉得远长于单次请求。只要你在复用线程的环境里使用它，就要默认自己面对的是
一个需要严格回收的上下文容器，而不是一个写完就会自动消失的临时变量。
