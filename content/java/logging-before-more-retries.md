## 重试不是诊断手段

Java 服务一旦出现超时、偶发失败或者下游波动，很多人的第一反应是再包一层重试。
这当然有用，但它更像止血，不像诊断。没有足够的日志和指标支撑，重试只会把问题拖长，
还可能把原本可控的故障放大成资源争用。

真正排障时，先要回答的是失败发生在哪里、持续了多久、影响了多少请求，而不是先问“要不要重试三次”。

## 日志要能还原一次请求的路径

服务日志最怕两种情况：要么太少，什么都看不出来；要么太散，连起来像碎片。
比较实用的方式，是围绕一次请求记录清楚几个关键点：请求标识、核心参数、外部依赖、耗时分段和最终结果。

这样当线程池排队、数据库变慢或者 HTTP 调用抖动时，你才能沿着同一条链路把问题串起来。
没有路径感的日志，只能证明“出过错”，不能解释“为什么错”。

## 指标要先抓住趋势，再看个例

单条异常很容易误导判断，真正有价值的是趋势。比如：

1. 请求量是否突然下降或上升。
2. P95、P99 延迟是否拉高。
3. 错误率是集中在某个接口，还是全局扩散。
4. 线程池、连接池、队列长度有没有同步异常。

这些指标合在一起，往往比一条报错栈更能说明系统状态。因为多数线上问题不是单点故障，而是多个小瓶颈一起叠出来的。

## 错误语义要比异常堆栈更稳定

如果服务对外只抛一堆原始异常，排障的人每次都得先翻栈追根因，效率会很差。
更好的做法，是把常见错误分类成稳定的语义，比如参数问题、依赖超时、资源不足、状态冲突、幂等冲突。

这样无论是监控告警、日志检索还是接口调用方处理，都能更快知道该重试、该降级，还是该直接失败。

## 先把可观察性做好，重试才有意义

重试不是不能用，而是要建立在你真的知道出了什么问题之上。日志告诉你请求怎么走的，
指标告诉你系统哪里开始偏了，错误语义告诉你这次失败该怎么处理。三者补齐之后，
重试、降级、熔断才不只是“碰运气的恢复动作”，而是有边界的治理手段。
