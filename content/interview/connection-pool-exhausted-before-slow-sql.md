## 不要一看到超时就直接怪数据库

线上最常见的报错往往是这样的：应用日志里先出现 `getConnection timeout`，业务接口开始大量失败，
但 DBA 侧暂时还没有看到明显的慢 SQL。很多人这时候会下意识认为监控有延迟，或者数据库马上就要崩了。
这种判断并不稳，因为连接池打满的触发条件比“数据库执行慢”更宽。

连接被借出去以后，只要迟迟回不来，池子一样会被耗尽。它可能卡在 SQL 执行，也可能卡在事务提交前的业务逻辑、
外部 RPC、消息发送，甚至是代码里漏关连接。也就是说，连接池告警反映的是“连接周转变慢”，
不是自动等价于“数据库语句已经慢到肉眼可见”。

## 第一层先看：连接到底是借不出来，还是借出去回不来

排查时先把问题分成两个方向：

1. 新请求拿不到连接，说明池中空闲连接不足。
2. 空闲连接不足，通常说明已有连接释放得太慢。

所以第一步不是马上抓最慢 SQL，而是看连接池运行指标：`active`、`idle`、`pending`、连接等待时间、超时次数。
如果 `active` 长时间顶满、`idle` 接近零，而请求线程持续堆积，说明连接都被占住了；接下来要继续问，
这些连接是在数据库里执行，还是已经拿到连接却在应用代码的别处停住。

## 第二层先查事务边界是不是太长

真实项目里，一个很常见的坑是事务包得过大。代码进了 `@Transactional` 方法后，先查库、再做参数组装、
然后调远程服务、写审计日志、发消息，最后才提交。数据库本身可能并没有慢到出慢 SQL，
但连接和事务会在整个方法结束前一直占着。

这类问题通常有几个信号：

1. SQL 单条执行时间并不夸张，但接口整体耗时明显变长。
2. 线程栈里能看到业务线程拿着连接等待 HTTP 或 RPC 返回。
3. 数据库连接数很高，但慢查询列表并不长。

如果出现这种情况，优化重点就不是给连接池继续加容量，而是收紧事务边界。能放到事务外的远程调用、
日志记录、异步通知，就不要让它们长期占着数据库连接。

## 第三层排查是不是连接泄漏或归还延迟

如果事务范围看起来正常，就该怀疑连接有没有被正确归还。尤其是历史代码混用 ORM、`JdbcTemplate`、
手写 JDBC 或多数据源时，最容易出现异常分支没有关闭资源的问题。

这时可以重点看三件事：

1. 连接池是否开启了泄漏检测或长时间占用日志。
2. 错误日志里是否存在拿到连接后抛异常但没有进入统一清理的分支。
3. 最近是否改过批处理、导出、定时任务这类长生命周期逻辑。

很多连接泄漏并不会立刻把数据库压高，而是随着请求积累，慢慢把池子榨干。它的外在表现就是：
业务高峰一来就先报连接池超时，但数据库 CPU 和慢 SQL 监控一开始并不明显。

## 第四层再回头确认数据库有没有隐形阻塞

慢 SQL 不明显，不代表数据库完全没问题。还有几类情况会让连接“看起来不慢，但就是回不来”。

比如：

1. 行锁或间隙锁等待时间长，但最终执行时间还没长到进入慢日志阈值。
2. 连接建立和认证正常，真正卡住的是提交阶段的锁竞争。
3. 主从切换、网络抖动或连接重置让一部分请求重试，导致连接占用时间拉长。

所以数据库侧至少还要看锁等待、活跃会话、事务持续时间和连接数趋势，不能只看一张慢 SQL 报表。

## 面试里怎么回答更像有实战经验

如果被问“连接池先满了，但慢 SQL 还没出来，通常该先查什么”，一个更稳的回答可以是：

先确认连接池是空闲连接耗尽，还是请求端突然激增；然后检查连接占用时长和事务边界，判断是不是应用拿着连接做了过多非数据库工作；
再排查连接泄漏、异常分支未释放资源，以及数据库中的锁等待和长事务；只有确认瓶颈确实在数据库执行本身之后，
再去重点优化 SQL 或扩容连接池。

这个回答的关键不是一句“先查数据库”，而是说明你知道连接池问题本质上是连接周转问题，应用和数据库两边都可能是根因。
