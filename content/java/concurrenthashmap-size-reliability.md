## 很多人把 `size()` 当成了“当前真相”

面试或代码评审里，经常能见到这样的写法：先判断 `map.size()` 是否超过阈值，再决定是否继续放数据、
是否触发告警，或者是否启动某种补偿逻辑。写的人往往默认 `size()` 代表“这一刻容器里到底有多少元素”，
但在 `ConcurrentHashMap` 里，这个假设并不牢靠。

原因不复杂。`ConcurrentHashMap` 的核心目标是让高并发读写尽量少互相阻塞，因此它不会为了给你一个绝对精准的
瞬时大小，就把所有更新操作串行化。既然写入和删除都可能在多个线程里同时发生，那么你读到的 `size()`，
更接近某个时刻的统计结果，而不是适合作为强一致业务判断的“现场清点”。

## 为什么并发容器不愿意为精确大小付高成本

如果一个普通 `HashMap` 只在单线程里使用，那么维护一个精确 `size` 很简单：每次 `put` 新键时加一，
每次 `remove` 成功时减一，读取时直接返回就行。

但换到并发场景，问题就来了。多个线程可能同时插入、删除、覆盖值，甚至还伴随扩容迁移。
这时如果想让 `size()` 在任何瞬间都绝对精确，就需要让所有涉及计数的操作在同一个强同步机制下执行。
做是能做，但代价通常是吞吐下降、锁竞争上升，最终背离了并发容器设计的初衷。

`ConcurrentHashMap` 采取的是更工程化的思路：让更新尽量分散，让计数尽量低冲突，再在读取大小时把这些计数汇总。
这使它非常适合“高并发下大致知道规模”的场景，却不适合把 `size()` 当成精确门槛。

## JDK 里的计数思路，本质上是在平衡性能和准确度

从实现思路上看，`ConcurrentHashMap` 并不是只有一个全局计数器。为了降低竞争，它会采用分散计数的方式：
低冲突时直接更新基础计数，竞争激烈时把压力摊到多个计数单元，再在读取时做汇总。

这样的好处是明显的：

1. 频繁写入时，不容易所有线程都卡在同一个计数热点上。
2. 读取大小不需要长期阻塞所有写线程。
3. 容器整体吞吐更稳定，符合并发集合的定位。

但副作用也同样明确：当别的线程正在持续修改 map 时，你拿到的 `size()` 只是一份尽力而为的聚合结果。
对监控统计、容量观察来说，这通常够用；对“如果已经满 1000 条就绝不能再写”这种业务约束来说，就不够稳。

## 典型误用：把 `size()` 当限流开关

最常见的误区，是用下面这种模式控制容量：

```java
if (cache.size() < 1000) {
    cache.put(key, value);
}
```

看起来很自然，实际上在并发下并不成立。线程 A 判断时 `size()` 可能是 999，线程 B 在同一时刻也看到 999，
于是两边都继续写，最终实际数量就可能超过 1000。更糟的是，如果中间还有删除、覆盖或扩容，结果会更不可预测。

这里的问题不是 `ConcurrentHashMap` “算错了”，而是调用方把一个弱实时统计值，用成了强约束判定条件。
容器从来没承诺帮你完成这层原子性。

## 什么时候 `size()` 是可以用的

也不能因此得出“`size()` 完全不能用”的结论。它在很多场景下依然很合适，比如：

1. 打日志，观察缓存、会话、任务表的大致规模。
2. 监控上报，辅助判断增长趋势是否异常。
3. 在非强一致链路里做粗粒度分桶或经验性调优。

这些场景关注的是趋势和数量级，而不是某一瞬间差 1、差 2 就会引发错误行为。
只要把预期放对，`size()` 仍然是一个有价值的指标。

## 需要严格上限时，应该把约束前移

如果业务真的要求“不能超过某个精确数量”，就不要把希望寄托在 `ConcurrentHashMap.size()` 上。
更合理的方案通常有三类：

1. 用额外的原子计数器配合明确的写入协议，保证判断与更新属于同一个受控流程。
2. 直接使用带容量语义的组件，比如阻塞队列、信号量或专门的缓存框架。
3. 把约束放到更高一层的单线程 actor、分片管理器或数据库唯一约束里处理。

核心原则是一样的：需要强约束，就要显式设计强约束，不要把并发容器的便利接口误当成业务事务边界。

## 面试里更值得说的是“为什么不能依赖它”

如果面试官问“`ConcurrentHashMap` 的 `size()` 准不准”，很多人会立刻开始背 `baseCount`、`counterCells`、
重试次数和遍历细节。会这些当然不坏，但更重要的是先把工程结论说清楚：

`size()` 在并发修改期间不适合做实时精确判断，因为容器为了吞吐和低竞争，采用了近似聚合计数的策略。
如果业务需要严格限制数量，应该通过更强的同步协议或更合适的数据结构来保证。

这比单纯背源码更能体现你理解了设计取舍。源码细节是“它怎么做到的”，而场景边界是“我们该怎么正确使用它”。

## 最后落到一句实战建议

写并发代码时，最好习惯性问自己一句：我此刻读到的是“绝对真相”，还是“足够有用的观测值”？
`ConcurrentHashMap.size()` 明显属于后者。它擅长告诉你系统大概处在什么规模，不擅长帮你完成必须一步不差的业务决策。

把这个边界想清楚，很多并发 bug 其实在写下第一行判断之前就已经能避开了。
