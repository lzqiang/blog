## 事务一超时，先别急着把数字往上拧

Java 服务在线上出现事务超时，并不算少见。
日志里一旦开始连续出现 `TransactionTimedOutException`、锁等待超时或者接口整体耗时拉长，
很多团队的第一反应都是先改配置：原来 3 秒不够，就调到 10 秒；10 秒还不稳，就再往上抬。

这个动作不能说完全没用。
如果最近只是流量短时波动，或者某个批量任务把数据库拖慢了一阵，适当放宽阈值的确可能帮系统避开一轮误伤。
但如果事务本身已经持续变慢，单纯调大超时时间，本质上只是让一次失败拖得更久。
原来 3 秒就暴露的问题，被藏到 10 秒、20 秒之后，线程、连接和锁却会被占得更久，后果往往比超时本身更难看。

所以遇到事务超时，真正该先问的不是“超时值要不要改”，而是“这段事务为什么需要这么久”。

## 超时只是结果，真正的问题常在事务边界里

很多人排查事务问题时，只盯着数据库执行慢不慢。
但一个 Spring 事务真正包住的，往往不只是单条 SQL。
它里面可能同时包含：

1. 参数校验和业务判断。
2. 多次数据库读写。
3. 远程 RPC 或 HTTP 调用。
4. 消息发送、缓存更新、文件处理。
5. 失败重试或补偿逻辑。

只要其中任何一步放进事务里，整段持锁和持连接的时间都会被一起拉长。
有些团队明明只是某个远程接口偶发抖动，最后却表现成数据库事务超时；
也有些系统真正慢的是一段循环更新，但大家只看见最外层报了事务异常。

所以超时不是起点，它只是提醒你：事务边界里混进了不该放进去的慢动作，或者本来就存在的慢 SQL 已经开始影响整体链路。

## 为什么把超时调大，经常会让问题从“报错”变成“拖垮”

超时时间调大之后，最直观的变化是报错变少了。
这很容易让人误以为问题已经缓解。
可从系统资源角度看，事情常常正往相反方向走。

假设一个接口事务里有两次更新和一次外部库存校验。
库存服务最近 P99 从 200 毫秒涨到了 4 秒，原来的事务超时配置是 5 秒，于是接口开始偶发失败。
这时候把超时改成 15 秒，短期看确实可能“恢复成功率”，
但代价是每个慢请求都会更长时间占住数据库连接、线程池工作线程和相关行锁。

接下来很容易出现几个连锁反应：

1. 连接池排队时间上升，其他请求也开始变慢。
2. 行锁持有更久，冲突事务被迫等待。
3. 应用线程堆积，Tomcat 或业务线程池吞吐下降。
4. 上游因为接口变慢开始重试，进一步放大并发压力。

原来只是少量请求在 5 秒处失败，现在变成大量请求在 10 秒到 15 秒之间一起卡住。
从业务视角看，错误可能晚一点出现；从系统视角看，资源耗尽反而更快到来。

## 先拆耗时，别把整个事务当成一个黑盒

处理这类问题，最有价值的第一步通常不是立刻改参数，而是把事务内部耗时拆开。
至少要看清楚三件事：

1. 进入事务之前有没有已经很慢的前置逻辑。
2. 事务内部最重的是哪条 SQL、哪段循环还是哪次远程调用。
3. 提交阶段有没有锁竞争、刷盘等待或连接池获取过慢。

如果系统已经接了链路追踪，最好直接看一次典型慢请求的 span 分布。
没有完整追踪也没关系，给事务内的关键步骤打分段日志，一样能快速定位大头。

很多“事务超时”最后都会被拆成很不同的问题：

1. 某条查询没有合适索引，导致后续更新迟迟开始不了。
2. 批量处理写在一个大事务里，单次提交包得过重。
3. 事务里调用外部服务，网络抖动直接扩大持锁时间。
4. 同一热点记录频繁更新，锁等待把整体耗时拉高。

只有先拆开，后面的优化动作才不会拍脑袋。

## 事务里最该优先清理的，是那些“不必在事务内完成”的动作

我见过不少服务，事务超时的直接原因并不是数据库太弱，而是事务边界画得过大。
例如保存订单后立刻在事务里调用营销接口、计算推荐结果、写审计日志，甚至同步发通知。
这些动作一旦变慢，就会把真正需要原子性的数据库更新一起拖住。

更稳妥的做法通常是把事务只收敛到必须一致的那部分：

1. 只在事务里完成核心状态变更。
2. 把通知、埋点、非关键计算改到事务外。
3. 需要最终一致的动作通过消息或 outbox 机制异步处理。

这样做的好处不是“更高级”，而是能直接缩短连接和锁的占用时间。
事务越短，系统在高峰期越不容易因为少数慢调用产生雪崩式放大。

## 如果真是数据库慢，也要先分清是 SQL 慢还是竞争慢

有些事务超时确实主要出在数据库层，但数据库慢也不是一个单一问题。
至少要区分两类：

1. SQL 本身执行就慢，例如扫描行数太多、索引失效、排序回表代价高。
2. SQL 文本不算重，但因为锁等待、热点更新或连接池紧张而变慢。

这两类问题的处理方法完全不同。
前者更偏向执行计划、索引和改写语句；
后者则要看事务设计、更新顺序、热点拆分和并发控制。

如果不先分清楚，只凭“数据库慢了”这句话去调超时，常常只是把不同层次的问题都盖在一层配置下面。

## 什么时候可以调大超时，什么时候不该这么做

超时配置不是绝对不能改，但它应该是建立在定位之后的动作，不该先于定位。

比较合理的场景包括：

1. 已确认最近是短时流量高峰，核心 SQL 和锁等待都在可接受范围内。
2. 某个批处理或导入任务天生耗时略长，且执行频率可控。
3. 代码已经完成优化，只是阈值仍然低于实际稳定耗时，需要校准配置。

而下面这些情况，通常不该把“调大超时”当主解：

1. 事务里混有远程调用或大量非必要逻辑。
2. 锁等待已经明显上升。
3. 连接池接近打满。
4. 接口慢导致上游开始重试。

这时候继续调大，只会把本来局部可控的慢请求，放大成系统级拥塞。

## 更成熟的处理顺序，是先缩事务，再谈阈值

如果面前真的有一段经常超时的 Java 事务，我会更倾向于按这个顺序处理：

1. 先抓一批慢请求样本，拆出事务内的主要耗时段。
2. 清理事务里不需要原子性的动作，缩短边界。
3. 检查热点 SQL、锁等待和连接池指标，确认瓶颈在哪一层。
4. 对症优化 SQL、批量方式、更新顺序或并发控制。
5. 最后再根据真实稳定耗时，决定是否调整事务超时配置。

这个顺序看起来比“先改个参数试试”慢一点，
但它更接近线上系统真正需要的东西：不是把错误藏起来，而是把慢点拆出来。

## 超时时间是护栏，不是性能优化手段

事务超时配置的意义，更像一层护栏。
它告诉系统：一段原本应该很快完成的工作，拖到这个程度就不值得继续占资源了。
如果把护栏一再往外挪，却不处理路面本身的问题，车看起来也许暂时没撞上，
但整条路只会变得越来越拥堵。

所以看到事务超时，不妨先克制一下调整配置的冲动。
真正有价值的工作，通常不是把数字调大，而是把事务里的慢动作一个个找出来，
该移出去的移出去，该优化的优化掉。等事务本身足够短，超时值才只是校准项，而不再是遮羞布。
