date: 2026-07-23

summary: 真正可控的 Agent 不是一直往前跑，而是在信息不足、目标冲突或工具失效时及时停下并交回判断权。
---

## 先把“继续做”改成“先判断能不能做”

很多 Agent 设计一开始就默认模型应该把事情做完，于是流程里最重的部分变成了“怎么追问、怎么补齐、怎么兜底”。
问题是，输入一旦本来就不完整，继续跑下去通常不是更聪明，而是更会把错误包装成进展。

更稳的做法，是把第一步改成判断任务是否具备最基本的执行条件。比如目标是否明确、关键字段是否齐全、依赖工具是否可用，
以及这次动作是否真的允许自动完成。只要这些前提不成立，最好的结果不是“勉强输出”，而是尽快停止。

## 不清楚的时候，追问比猜测更省成本

Agent 最危险的地方，不在于它不知道，而在于它不知道自己不知道。很多流程看起来一路顺滑，
其实只是把缺失信息自动补成了一个看似合理的结果。

这类问题常见于工单分类、资料整理、内容生成和任务拆解。模型如果拿到的上下文不足，
应该先追问最少量的关键信息，而不是把空白全用推测填满。因为一次准确追问，通常比事后纠错便宜得多。

## 给流程加一个明确的停机条件

一个可控的 Agent，最好至少具备三种停止信号：

1. 目标无法唯一确定。
2. 工具调用连续失败。
3. 结果已经低于可接受置信度。

这些条件的作用不是限制能力，而是防止系统在错误路径上越走越远。真正成熟的自动化，应该能体面地停下来，
把决定权交回给人，而不是把“不确定”伪装成“已完成”。

## 可长期运行的系统，通常都不执着

我越来越觉得，Agent 设计里最重要的能力之一，是知道什么时候不该继续。能停，说明系统理解边界；
能转人工，说明系统承认现实；能在信息不足时拒绝硬答，说明它真的把质量放在了结果前面。

所以与其把流程改得更长，不如先把停止条件写清楚。很多线上事故，都是从“再试一次，也许就对了”开始的。
