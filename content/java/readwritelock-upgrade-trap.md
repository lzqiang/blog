## 锁升级会制造互相等待

很多人第一次使用 `ReentrantReadWriteLock` 时，都会自然地写出这样的流程：
先拿读锁检查状态，发现需要修改时，再在同一线程里申请写锁。问题在于，
读写锁允许多个读者并发持有读锁，而写锁必须等所有读锁释放后才能进入。

如果两个线程都先拿到读锁，再几乎同时尝试升级为写锁，就会出现经典僵局：
每个线程都在等对方先把读锁放掉，但大家又都不愿意先放，因为一旦释放，
刚才读到的状态就可能失效。JDK 没有为这种“升级”提供安全语义，本质上是
为了避免把一个看似方便的 API 包装成更隐蔽的死锁入口。

## 正确做法通常是释放后重试

更常见的写法是“读锁检查，释放读锁，再申请写锁，拿到后再次校验条件”。
这会多一次状态判断，但逻辑清晰，也符合写锁独占的设计。

例如缓存懒加载、配置热更新这类场景，第一次读取只是为了快速过滤掉大多数
不需要修改的请求；真正进入写路径时，应接受状态可能已经被别的线程改变，
因此二次检查是必须的，而不是“保险措施”。

如果你发现释放再重试导致代码层层嵌套，往往说明共享状态边界设计得太粗。
与其在一个大对象上同时承载查询和变更，不如拆成更小的状态单元，或者把
写操作集中到单线程执行，让读取路径保持无锁或低锁竞争。

## 什么时候考虑别的并发工具

`StampedLock` 提供了从乐观读到悲观写的另一套思路，但它也不鼓励直接锁升级，
而是通过校验戳记失效后重走流程。它适合读远多于写、且读取代码非常短的场景，
但可中断性、可重入性和易用性都不如 `ReentrantReadWriteLock`。

如果目标只是保护一个缓存 Map，`ConcurrentHashMap` 加 `computeIfAbsent`
往往比手写读写锁更稳妥；如果目标是把更新串行化，消息队列、Actor 模型或
单线程事件循环也可能更简单。并发工具的选择，不是看“谁功能更多”，而是看
谁最接近你的真实数据流。
