## 很多“自动重试”问题，根子不在重试本身

线上服务一旦开始偶发失败，团队通常会先想到加重试。这个反应没错，因为网络抖动、下游短暂超时、
瞬时资源争抢，确实都可能通过一次或两次重试自然恢复。问题在于，很多代码并不是在一个小而明确的远程调用上重试，
而是把整段业务方法一起包进重试框架里，甚至把数据库事务也一起包进去。

这样写的表面优点是省事：注解一加、循环一套，好像就把不稳定问题交给框架解决了。
但真正进入生产环境后，麻烦往往不是“重试没生效”，而是它生效得太粗了。一次失败本来只影响某个下游调用，
结果却把整段事务重新执行，连已经成功的本地校验、库存扣减、状态流转都被迫再来一遍。

所以讨论重试时，我越来越倾向于先问一个问题：你到底想重试哪一步？如果这个边界没划清，
后面大概率会在数据一致性、锁竞争和副作用控制上补更大的洞。

## 事务边界放得太大，失败成本就会被一起放大

设想一个典型下单流程：先查订单状态，再写本地数据库，然后调用支付或库存服务，最后提交事务。
如果整个方法外层有事务，外面再套一个“失败就重试三次”的机制，那么一次远程调用超时，
就意味着这整段事务逻辑会被重新跑。

这里至少有三个直接成本：

1. 每次重试都会重新占用数据库连接、锁和事务上下文。
2. 事务里已经完成的本地写操作会重复执行，需要额外依赖幂等约束兜底。
3. 一次本该局部恢复的问题，被扩散成多次完整业务执行。

在低并发环境里，这类问题不一定立刻爆出来，所以很多团队会误以为“这样也能跑”。但并发一上来，
你会发现锁等待时间变长、数据库冲突增多、日志里重复请求难以追踪，最后根本分不清是业务真的被调用了三次，
还是系统为了自救主动制造了三次。

## 远程调用可重试，不代表事务整体可重试

很多工程师第一次接触重试时，会自然地把两件事混在一起：

1. 某个外部依赖失败了，应该允许再试一次。
2. 当前业务方法失败了，那就把整个方法重新执行。

这两件事在语义上并不等价。前者重试的是一个不稳定步骤，后者重试的是整段业务副作用。

比如一个发券流程里，真正不稳定的也许只是调用消息平台发通知；用户资格校验、券状态写入、领取记录落库，
这些步骤本身并没有问题。如果把整个事务边界一起重跑，就意味着这些本地状态也会跟着重做。
哪怕数据库唯一索引能拦住重复写，系统依然承担了额外的锁冲突和执行开销。

更糟的是，有些步骤并不天然可重放。扣减库存、生成流水号、推进状态机、写审计日志，
这些动作只要有一个没有被设计成幂等，整段重试就可能把一次可恢复故障变成真实的数据问题。

## 更稳的做法，是把事务和重试拆成两个明确层次

相比“整段一起重试”，更可靠的设计通常是：

1. 先把本地事务尽量收缩，只负责维护本地一致状态。
2. 把真正可能短暂失败的远程调用单独抽出来，在事务外做有限重试。
3. 如果远程结果会影响最终状态，就通过状态机、补偿任务或 outbox 机制继续推进，而不是长事务硬等。

这样做的核心，不是为了让代码看起来更“分层”，而是为了把失败的影响面控制住。
本地事务成功，就先把本地可确定的事实落下来；外部系统暂时不可用，就按可恢复故障处理，
由重试、补偿或异步任务接手。两者职责分清后，系统行为会更容易解释。

例如订单创建成功后，通过 outbox 表记录“待通知库存系统”的事件，由异步消费者发送。
库存服务短时失败时，只重试发送动作，不重做订单创建事务。这样既能保留恢复能力，
又不会把核心库表拖进反复回滚和重复执行里。

## 如果事务里必须调用下游，就要先回答三个问题

现实项目里并不是所有链路都能立刻改成事务外异步推进。有些场景出于历史原因，
确实会在事务附近调用下游。这时至少要把三个问题回答清楚：

1. 失败后重试的最小单元是什么。
2. 已经执行过的本地或远程副作用，是否具备幂等保障。
3. 当重试次数耗尽时，系统留下的中间状态是否可观测、可补偿。

如果这三个问题里任何一个没有答案，那把整段逻辑套进重试，大概率只是把风险隐藏起来。
很多线上事故复盘都会走到同一个结论：代码里不是没有异常处理，而是异常处理过于乐观，
默认“再来一次就好”，却没有认真定义第二次到底在重复什么。

## Spring 里尤其容易踩到“注解叠注解”的坑

Java 项目里常见的写法，是在同一个方法上同时使用 `@Transactional` 和重试相关注解，
看上去几乎不费力。问题在于，注解把边界藏起来后，开发者更容易忽略调用顺序和代理行为。

有的人以为自己重试的是一次 HTTP 调用，实际上重试的是整个事务方法；有的人以为事务每次失败都会完整回滚，
却忘了外部系统已经收到了上一轮请求；还有的人在日志里只看到了“第三次成功”，
没意识到前两次已经对下游造成了两次压力。

所以比起依赖注解自动组合，我更建议把关键路径显式写出来：哪里开始事务、哪里结束事务、
哪里是可重试远程调用、重试条件是什么、失败后由谁接管。代码多几行不可怕，可怕的是边界不清时，
系统在悄悄替你做超出预期的事。

## 真正要优化的，不是“重试次数”，而是失败后的控制面

很多团队排查这类问题时，习惯先改两个参数：把重试次数从三次降到两次，或者把间隔从一秒改成三秒。
这些调优有时有用，但通常只是止痛，不是治本。更关键的是把系统设计成：

1. 知道哪些失败值得重试，哪些失败应该立即终止。
2. 知道哪些步骤可重放，哪些步骤必须保证只执行一次。
3. 知道失败后数据会停在哪个状态，以及谁负责把它推进到终态。

一旦控制面清楚，重试就会从“盲目再试一次”变成“对特定不稳定环节做有限恢复”。
这时即使保留事务，也不会轻易把整段事务边界一起拖下水。

## 把边界讲清楚，系统才会真的稳定

重试逻辑不要包住整个事务边界，不是因为事务和重试天然冲突，而是因为两者解决的问题不同。
事务负责局部一致性，重试负责处理暂时性失败。把它们简单叠在一起，通常只会让失败范围扩大、
副作用更难控制。

真正稳的系统，往往不是“哪里报错就整段重来”，而是能精确知道哪一步不稳定、哪一步必须只做一次、
哪一步失败后可以交给后续流程继续收拾。把这几个边界划清，重试才会成为修复能力，而不是新的故障来源。
