## 定时任务最容易让人低估的地方，是它看起来总是很安静

很多 Java 服务上线后，除了接口请求，还会挂着一批定时任务：同步对账、补偿重试、扫描超时订单、
刷新缓存、归档日志、生成日报。它们平时默默运行，日志里偶尔打几行“task started”、
“task finished”，很容易让人把它们归类成一种背景工作，觉得只要逻辑写对、Cron 配好，系统就会一直平稳。

真正出问题的时候，往往不是任务“不执行”，而是它在一个错误的时机和错误的一批请求抢占了同一套资源。
白天业务高峰来了，定时任务也刚好开始扫库；接口线程还在等数据库返回，任务线程又补了一轮远程调用；
最后你看到的是 RT 抖动、连接池紧张、下游告警，而根因却藏在一个平时没人盯的调度逻辑里。

## 为什么“共用线程池”在低负载时很省事，在高负载时却很危险

很多项目一开始图省事，会让定时任务直接复用已有线程池，或者干脆用 Spring 默认调度器跑所有任务。
这种做法在业务量小时不一定有问题，因为资源够用、任务也短，彼此之间几乎感觉不到竞争。

但只要任务里包含以下几类操作，风险就会上来：

1. 扫描数据库并按批处理大量记录。
2. 调用外部接口做补偿、同步或推送。
3. 对缓存、消息队列或对象存储做集中写入。
4. 某个失败任务会触发重试，形成短时间放大。

这些任务的共同点是，它们虽然不是用户主动发起，却会真实占用线程、连接、CPU 和网络带宽。
如果它们和在线请求共用一套执行资源，线上高峰期就会出现一种很尴尬的局面：
接口认为“自己只是正常流量”，任务也认为“自己只是按计划执行”，最终两边一起把系统推到边界上。

## 定时任务的稳定性，不只取决于能不能跑完，还取决于会不会拖垮别人

很多人讨论定时任务时，习惯只问两个问题：会不会漏执行，失败后能不能重试。
这当然重要，但工程上还要补第三个问题：它执行时，会不会影响在线链路。

比如一个每五分钟跑一次的补偿任务，每次扫两万条订单，逻辑本身没有 bug，最终也都成功补齐了。
从任务视角看，一切正常；可如果它每次启动都把数据库连接池占到只剩几个空闲连接，
接口请求就会开始排队，P99 响应时间会明显变长。这个时候，任务不是“坏了”，而是“太不顾场合”。

所以看定时任务质量，不能只看成功率，还要看它对共享资源的冲击曲线。一个成功率 100%、
但每次执行都把业务高峰搅乱的任务，不能算设计良好。

## 更稳的做法，是把调度和执行都做隔离

比较稳妥的设计通常分两层：

第一层是调度隔离。触发器负责“什么时候开始”，尽量轻，不在调度线程里做重活。
真正的业务执行交给独立线程池，避免某个慢任务把后续调度全部卡住。

第二层是资源隔离。不同性质的任务拆开，不要让全量扫描、外部补偿、报表生成和缓存刷新都挤在一个池里。
如果任务执行涉及数据库和 RPC，下游限流、超时和批大小也要单独配置，不要沿用接口链路的默认值。

这背后的逻辑很简单：定时任务不是一个“后台黑盒”，而是一种定期施加到系统上的负载。
既然它会制造负载，就应该像对待在线流量一样，给它边界、预算和隔离。

## 批处理任务尤其要警惕“一次全扫完”的冲动

很多任务出问题，不是因为没有线程池，而是因为任务本身的工作模型太粗。
最常见的写法是定时拉起一轮，全表或大范围扫描，然后在内存里组装、过滤、逐条处理。
代码从功能上可行，但一到数据量上升阶段，数据库压力、GC 和执行时长会一起上来。

更合适的思路通常是：

1. 按时间窗口或主键范围分批拉取。
2. 每批限制条数，并记录游标或断点。
3. 为慢下游设置并发上限，不让任务自己把自己打爆。
4. 明确单次运行预算，必要时允许本轮只处理一部分。

很多人担心“这轮没扫完会不会积压”，可现实里更危险的往往是“为了扫完，把线上一并拖慢”。
定时任务允许延后一点，在线请求通常不允许。

## 监控上要单独看任务，而不是把它们埋在普通日志里

如果任务和接口共用监控口径，排查时会很痛苦。更好的做法是把以下信息单独打出来：

1. 每个任务的启动频率、运行时长和成功失败次数。
2. 每次处理的数据量、分批次数和重试次数。
3. 任务线程池队列长度、活跃线程数和拒绝情况。
4. 任务执行期间数据库连接池、下游调用 RT 的变化。

这样当线上出现波动时，你才能快速判断：是业务流量自己上来了，还是某个任务恰好在同一分钟开跑，
并且把共享资源压到了临界点。没有这层可观测性，很多任务问题最后都会被误判成“系统偶发抖动”。

## 最后落回一句实用判断

定时任务不要默认共用一个线程池，不是因为“线程池复用不优雅”，而是因为任务负载和在线流量的目标天然不同。
前者强调按批完成和最终收敛，后者强调即时响应和延迟稳定。把它们混在一起，低峰期看不出问题，
高峰期就会互相拖累。

如果你在项目里发现某个任务已经开始影响接口 RT、连接池水位或下游配额，别只优化那几行业务代码，
先回头看它是不是该被隔离出去。很多线上抖动，真正该拆的不是逻辑，而是资源边界。
