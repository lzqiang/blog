## 先别把答案收缩成一句“offset 越大越慢”

面试里问深分页为什么慢，是一个很常见的数据库题。很多人会直接回答：“因为 `limit offset, size`
前面的数据也要扫描，所以 offset 越大越慢。”这句话方向没错，但如果只停在这里，回答会显得有点薄，
更像背结论，而不是理解数据库到底慢在什么地方。

更好的回答应该再往下拆一层：数据库不是因为看见 `offset=100000` 这串数字就天然变慢，
而是因为它为了返回后面的那几条数据，通常必须先把前面大量记录找出来、排序或定位出来，
再把其中绝大部分丢掉。慢，慢在扫描成本、排序成本、回表成本，以及这种分页方式和业务目标之间的错配。

## 深分页的核心问题，是“取很多，再扔很多”

以 MySQL 为例，假设有一条 SQL：

```sql
select id, title, created_at
from article
where status = 1
order by created_at desc
limit 100000, 20;
```

业务真正想要的只是第 100001 到 100020 条数据，但数据库往往需要先定位出前 100020 条符合条件的记录，
然后跳过前 100000 条，只把最后 20 条交给客户端。也就是说，返回结果很少，不代表中间过程工作量也少。

如果排序字段和过滤条件没能很好命中索引，数据库还可能先做大范围扫描，再排序，再截取，成本会更高。
所以面试里如果只说“offset 大了”，还不够；你要说明数据库是在做大量无效工作。

## 为什么有时看起来走了索引，还是会慢

很多人接着会补一句：“建索引就好了。”这也不完整。深分页的麻烦恰恰在于，
即使走了索引，也不一定代表这条 SQL 就便宜。

原因通常有两类：

1. 索引只能帮助更快找到有序记录，但 `offset` 很大时，前面大量索引项依然要被遍历。
2. 如果查询列不被索引完全覆盖，定位到索引项后还要回表取数据，前面那些被跳过的记录也可能产生大量随机 IO。

这就是为什么有些 SQL 的 `explain` 看上去并不差，线上却依然在大页码时明显变慢。它不是完全没用上索引，
而是即便用了，代价仍然随着页码增大而快速放大。

## 真正值得讲的，是 offset 分页适合什么，不适合什么

面试官通常不只是想听你解释数据库原理，也想看你会不会把问题拉回业务设计。
`limit offset` 这种分页方式并不是一无是处，它对后台列表、小数据集、偶发翻页其实很方便，
实现简单，前端也好理解。但当数据量很大、页码很深、访问频繁时，它的成本就会变得不划算。

这时更值得讨论的是：这个场景到底需不需要“跳到第 5000 页”这种能力？
很多用户行为其实并不会真的翻到那么后面。若业务只是做“加载更多”或“按时间向后看”，
那游标分页、基于主键或时间戳的 seek 方法，通常比 offset 分页更稳。

也就是说，深分页问题很多时候不是“数据库参数没调好”，而是“分页方案选得不合适”。

## 面试里可以顺势讲游标分页，但别把它说成银弹

一个更完整的回答，通常会自然过渡到游标分页。比如按 `created_at` 和 `id` 这类有序字段继续往后查：

```sql
select id, title, created_at
from article
where status = 1
  and (created_at < ? or (created_at = ? and id < ?))
order by created_at desc, id desc
limit 20;
```

这种方式的好处，是每次都从上一次结果的边界继续取，而不是每次都从头数到第几十万条。
数据库不需要做大规模跳过动作，性能通常会稳定很多。

但这里也别回答得过于理想化。游标分页的代价是不能轻易随机跳页，对前端交互和业务体验有要求；
排序字段还必须稳定且可比较，否则也容易出现重复或遗漏。所以更稳的表达是：
当业务主要是顺序浏览时，游标分页更合适；当确实需要任意跳页时，就要接受 offset 模式的成本，
或者考虑额外的搜索、筛选、归档策略，减少深翻页需求。

## 如果面试官继续追问，可以从执行计划和数据分布展开

更进一层的回答，可以提到你会怎么验证问题：

1. 用 `explain` 看是否命中过滤和排序索引。
2. 看查询列是否能被覆盖索引承接，减少回表。
3. 看数据分布是否导致过滤选择性很差，索引虽然存在但效果有限。
4. 结合慢 SQL 日志判断是排序、扫描还是回表成本在放大。

这样回答的好处，是你不仅知道“为什么慢”，也知道上线排查时该看什么证据。
面试题最容易失分的地方，就是只会给概念，不会给验证路径。

## 最后用一句话收束更稳

面试被问深分页为什么慢时，不要只回答 `limit offset` 前面也会扫描，
而要进一步讲清楚数据库是在“取很多、扔很多”，并且随着 offset 增大，扫描、排序和回表成本都会被放大。
再往前一步，说出什么时候该继续优化索引，什么时候该直接重做分页方案，这个回答就会明显更完整。
