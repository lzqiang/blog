## 先别急着说“数据库性能差”

面试里一旦提到写入变慢，很多人会很快把原因归到数据库本身，比如磁盘慢了、CPU 打高了、实例规格不够。
这些可能性当然存在，但真实线上环境里，更常见的情况其实是业务写法先把数据库推到了低效状态。

尤其在订单、库存、账务、日志归档这类写入密集场景中，问题通常绕不开两个方向：
一类是并发写入之间互相等待，表现为锁竞争明显；另一类是每次写入都太碎，批量方式设计得不好，
导致数据库一直在高频做重复的网络往返、事务提交和索引维护。

所以如果面试官问“数据库写入变慢时，先看锁竞争还是批量方式”，更稳妥的回答不是二选一，
而是先建立判断顺序。

## 第一层判断，看慢在等待还是慢在执行

排查这类问题时，我会先区分写请求到底是“执行本身很慢”，还是“花了很多时间在等”。

如果应用侧大量请求 RT 变长，但数据库 CPU 并不高，连接数却居高不下，同时活跃会话里出现明显的 lock wait、
row lock contention 或事务阻塞链，那优先级就该放在锁竞争上。因为这说明 SQL 也许并不重，
但并发事务互相卡住了，吞吐自然上不去。

反过来，如果数据库几乎没有明显锁等待，单条 insert 或 update 自身执行并不夸张，可整体 TPS 还是上不来，
而应用层又是成千上万次小事务不断提交，那问题往往更接近批量方式不合理。数据库不是被“堵住”，
而是被大量细碎操作消耗掉了吞吐。

这一步很重要，因为锁问题和批量问题的优化路径完全不同。一个要先减少互相等待，一个要先减少操作次数。

## 锁竞争通常藏在更新热点和长事务里

如果判断更像锁问题，我会继续看两个典型场景。

第一种是热点行更新。比如多个请求同时修改同一条库存记录、同一笔账户余额、同一个任务状态，
哪怕每条 SQL 本身很简单，只要并发集中在少数记录上，锁等待就会快速叠起来。很多系统吞吐下降，
不是因为数据库处理不过来，而是因为大家都在排队等同一把锁。

第二种是长事务。业务里常见的坑是，在一个事务里先查很多数据、再做一堆计算、再调用外部服务，
最后才提交更新。这样事务持锁时间会被拉长，后面的写请求只能持续等待。表面看是“写入慢”，
本质上是事务边界太大，把锁占用时间拖长了。

面试中如果能补一句“要结合慢事务、阻塞链、热点更新记录一起看”，通常会比单纯说“检查锁”更像真实排障。

## 批量方式不合理，数据库会被大量小动作拖垮

另一类问题是没有明显阻塞，但写入方式天生低效。最常见的表现是应用程序循环单条插入：
每来一条数据就独立开启事务、发送一条 SQL、等待响应、提交一次。数据量一上来，网络往返、
日志刷盘、索引更新和事务提交成本都会被放大。

这时候即使数据库实例没有异常，也会显得“怎么都快不起来”。因为真正限制吞吐的不是单条 SQL 的复杂度，
而是写入粒度太小。

更合理的方式通常包括：

1. 合并多条写入，使用批量 insert 或批处理提交。
2. 控制单批大小，避免一批过大反而导致回滚成本和锁持有时间升高。
3. 区分实时强一致写入和允许异步落库的写入，别把所有数据都按最高实时要求处理。

这里的关键不是一味追求“批越大越好”，而是在事务大小、失败重试成本和数据库承载之间找平衡。

## 面试里更完整的回答方式

如果我来回答这个问题，会这样组织：

先判断写慢的时间消耗在等待还是执行。如果大量会话在等锁、热点更新明显、长事务较多，那优先看锁竞争；
如果锁等待不突出，但应用采用高频单条写入、小事务反复提交，整体吞吐上不去，就该优先优化批量写入方式。

然后再补一层：

锁问题重点优化事务边界、热点拆分、更新顺序和索引命中；批量问题重点优化提交粒度、批处理策略、
异步削峰和网络往返次数。只有先区分瓶颈在哪一层，优化动作才不会打偏。

## 真正稳的判断，是把现象和机制对应起来

数据库写入变慢，不适合靠经验口号直接拍结论。你需要把现象对应到机制：
是在等别人释放锁，还是自己每次只做一点点、却做了太多次。

所以最稳的一句话是：先看等待特征，再看写入粒度。能把这两层说清楚，面试官通常能判断你不是只背过几个优化名词，
而是真的理解并发写入为什么会慢，以及慢在什么地方。
