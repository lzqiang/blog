## 线上调线程池，最怕一开始就把公式当答案

很多人第一次认真调线程池，都会很快遇到那条经典公式：  
线程数 = CPU 核数 * 目标 CPU 利用率 * (1 + 等待时间 / 计算时间)

这条公式不是错的，它提供了一个理解入口：如果任务经常在等待外部资源，就可以让更多线程同时存在，
用等待时间去覆盖 CPU 空档。但真正到了线上，问题往往不是“有没有公式”，而是你根本不知道自己的等待时间
在等什么，计算时间又占了多少。

如果没有先搞清任务类型，只是看到线程池打满就加核心线程数，最后很容易把问题从“吞吐上不去”调成
“应用整体抖动更大”。线程多了，排队不一定少；上下文切换多了，CPU 反而更忙；下游接口本来已经慢，
你再放更多并发过去，只会把故障面一起放大。

所以我更倾向于把那条公式放到第二步。第一步应该是先判断：这个线程池里的任务，大部分时间到底花在什么地方。

## 先分三类：算得多、等得多、卡在外部约束

实践里，线程池里的任务大致可以分成三类。

第一类是计算型任务。比如本地加解密、规则计算、复杂 JSON 转换、图片处理这类逻辑，
线程运行时主要在消耗 CPU，等待外部资源的时间很少。这样的线程池如果开得太大，
收益通常并不明显，因为 CPU 本身就是瓶颈，额外线程只会带来更多切换开销。

第二类是 IO 型任务。比如调用远程服务、访问数据库、读写对象存储，线程的大量时间花在等待网络或磁盘返回。
这时候线程数通常可以比 CPU 核数多不少，因为一部分线程在等结果，CPU 并不会一直忙满。

第三类最容易被误判，就是“看起来像 IO，实际上卡在外部约束”的任务。比如下游接口有严格 QPS 限流，
数据库连接池本来就快满了，或者某个依赖的响应时间一到高峰就剧烈波动。这时线程池里线程虽然也在等待，
但你增加线程并不能换来更高吞吐，只会让更多请求同时堵在同一个瓶颈前面。

真正有用的判断不是“它是不是发了 HTTP 请求”，而是“线程等待之后，系统整体有没有机会换来更高完成量”。
如果没有，调大线程池只是在扩大拥塞面。

## 别急着定参数，先看四个观测面

我一般不会在没有观测的情况下直接给线程池改参数。至少先看四个东西。

第一，看活跃线程数是不是长期贴着上限。如果线程池峰值偶尔打满，但任务很快消化掉，
这更像是瞬时波动，不一定值得大改。如果活跃线程、队列长度、任务等待时长长期一起走高，
才说明池子真的承压。

第二，看任务执行时间和排队时间。很多系统只统计执行耗时，却不统计任务在队列里等了多久。
但对于线程池问题来说，用户真正感知的是“从提交到完成花了多久”。有时执行只要 50 毫秒，
排队却已经 2 秒，这时问题重点不是业务代码本身，而是调度能力不够或者任务混跑太多。

第三，看下游资源是否已经接近极限。数据库连接池使用率、外部接口 P95、Redis 超时、第三方限流返回码，
这些都要一起看。否则你可能把一个本来只是下游变慢的问题，误判成自己线程不够。

第四，看线程池里是不是混进了不同性质的任务。最常见的是把短平快的校验任务和重 IO 的同步任务丢进同一个池。
这样一来，慢任务一多，快任务也会被堵住。很多时候不需要先调参数，先拆池子就能明显改善。

## 计算型任务，通常先保守

如果确认线程池主要跑计算型任务，我通常会先保守配置，核心思路是别让线程数远远超过 CPU 能稳定承载的范围。

原因很直接。CPU 密集型任务并不会因为你多开很多线程就更快完成，反而容易因为频繁切换、缓存失效和竞争加剧，
让单位时间内真正做成的事更少。尤其是 Java 服务里，如果这些任务本身还会带来较高对象分配，
线程数过多还会把 GC 压力一起放大。

这类池子更值得关注的是：

1. 单个任务有没有不必要的重复计算。
2. 有没有热点数据可以提前预处理。
3. 是否可以拆批、降采样，或者把低优先级任务异步化。

参数只是最后一层。算法和任务模型如果不改，线程池再怎么调也只是局部止痛。

## IO 型任务，不是越大越好，而是越清楚越好

如果任务主要在等待 IO，线程数确实可以适当放宽，但这里最容易犯的错误是“因为大部分线程在等，所以继续加总没错”。

问题在于，IO 等待的背后可能对应完全不同的约束：

1. 数据库连接池是否足够。
2. 下游服务是否允许更高并发。
3. 单机内存是否能承受更多挂起请求。
4. 超时时间是否过长，导致失败请求占坑太久。

例如一个线程池负责调用外部结算服务，平均响应 80 毫秒，偶发高峰到 2 秒。
这时候直接扩线程池，短期也许能多吞一些请求，但如果高峰来自下游拥塞，
你只是把更多请求更快地堆到了对方门口。等超时一起回来，本地线程、连接、重试流量就会同时炸开。

更稳妥的做法通常是配合做三件事：

1. 给线程池设置明确的队列上限，避免无限堆积。
2. 给调用设置合理超时，别让无效等待持续占住线程。
3. 在必要时加限流、隔离和熔断，不让线程池变成故障放大器。

所以 IO 型线程池真正需要的不是“更大”，而是“知道自己在为谁等待，又能承受多大的等待”。

## 队列长度不是缓冲万能药

另一个常见误区是，不敢加线程，就先把队列调大一点，觉得这样至少不会立刻拒绝任务。

短期看，队列确实能吸收突发流量；长期看，过大的队列经常会掩盖问题。因为它把“处理不过来”的信号推迟暴露了，
服务表面上没有报错，但请求端到端延迟已经被悄悄拉长。等队列真正溢出时，系统往往已经积累了很多老请求，
这时即使开始拒绝，也很难快速恢复。

如果业务对时效性敏感，我反而更愿意把队列设得清楚一些，让系统尽早表达“现在接不住了”，
再配合降级、快速失败或回退逻辑。这样至少故障是可见的，而不是把问题藏在几千条待处理任务后面。

线程池不是仓库，不能默认用来长期囤积工作。它更像一道闸门，用来控制当前这台机器在这一类任务上愿意承担多少并发。

## 参数最终要靠回压思维收敛

我现在看线程池调优，越来越少把它当成一个“把数字调对”的问题，而更像是在设计一套回压机制。

线程数决定了同时有多少任务在跑，队列决定了系统愿意暂存多少压力，拒绝策略决定了超出能力时如何表达失败。
这三者合起来，才是一个线程池真正的行为边界。

如果没有回压意识，常见结果就是：

1. 线程数越调越大。
2. 队列越放越长。
3. 超时越设越久。
4. 最后把一个本该快速失败的小拥堵，放大成全链路雪崩。

反过来，如果先搞清任务的等待比到底来自哪里，再配合监控和压测逐步收敛参数，
线程池就会从一个“出问题再碰的配置项”，变成你主动控制系统负载形状的工具。

## 先认识任务，再谈公式

那条线程池公式依然有价值，它能帮助我们建立“等待多时可以提高并发”的直觉。
但真正决定参数是否合理的，不是你有没有把公式背下来，而是你是否理解这些线程每天到底在忙什么、
等什么、替谁承压。

先认识任务，再谈公式，线程池调优才不会一上来就走偏。
