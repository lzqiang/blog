## 先说结论：大多数普通临界区，`synchronized` 已经够用

很多人一被问到这个问题，就会立刻背出“`ReentrantLock` 更灵活、`synchronized` 更简单”。
这句话不算错，但如果只停在这一层，面试官通常听不出你是不是真的用过。

更稳的回答方式是先给出判断基线：如果只是保护一个很小的共享临界区，没有超时获取、可中断获取、
多个条件队列或公平性这类额外诉求，优先用 `synchronized`。它语义直接，代码短，出错面更小，
而且现代 JVM 对它的优化已经很成熟。

换句话说，不是 `ReentrantLock` 天生比 `synchronized` 高级，而是它提供了更多控制能力，
代价是开发者也要自己承担更多正确性责任。

## 两者最核心的区别，不是“谁性能更高”

早些年很多资料喜欢把这个问题讲成性能对比题，但在今天的 Java 面试里，这个角度已经不够可靠了。
JVM 对锁实现持续做过优化，轻量级竞争场景下，单纯说哪个一定更快，往往没有普适结论。

真正有区分度的，是以下几个能力差异：

1. `synchronized` 依赖 JVM 管理加锁和释放，退出同步块会自动释放锁。
2. `ReentrantLock` 需要显式 `lock()` 和 `unlock()`，通常必须配合 `try/finally`。
3. `ReentrantLock` 支持 `tryLock()`、可中断获取和多个 `Condition`。
4. `synchronized` 写法更收敛，可读性和维护成本通常更低。

所以面试时最好把“控制能力”和“代码风险”一起说，而不是只强调功能丰富。

## `ReentrantLock` 更适合需要“控制获取过程”的场景

什么时候我会优先考虑 `ReentrantLock`？通常是我不只关心“能不能加上锁”，还关心“如何加上这把锁”。

比如：

1. 线程不能一直傻等，需要 `tryLock()` 在获取失败后快速返回。
2. 任务可能要响应取消，希望在线程阻塞等锁时支持 `lockInterruptibly()`。
3. 一个锁对象下有多组等待条件，需要把等待队列拆开，而不是全放在同一个监视器上。
4. 需要更明确地观察排队、尝试获取和超时退让这些行为。

例如一个订单补偿任务线程，如果拿不到锁就应该直接放弃本轮，而不是无限等待，
这时 `tryLock()` 往往比 `synchronized` 更贴合业务语义。

## `synchronized` 的优势是“默认正确”

很多并发 Bug 不是因为程序员不知道高级 API，而是因为大家高估了自己手动管理锁的稳定性。
`ReentrantLock` 最大的风险，不是不好用，而是太容易因为分支返回、异常路径或重构疏忽漏掉 `unlock()`。

`synchronized` 在这方面更像“约束更强的安全栏”：

1. 进入同步块即加锁。
2. 正常退出或异常退出都会自动释放。
3. 阅读代码时，锁的作用域通常一眼能看清。

这也是为什么团队协作里，很多基础业务代码宁可选 `synchronized`。不是它能力最强，
而是它更不容易把简单问题写复杂。

## 条件队列是一个很值得说出来的差异

如果面试官追问更深一点，`Condition` 往往是个加分点。`synchronized` 配合 `wait()` 和 `notifyAll()`
也能做线程协作，但一个对象监视器本质上只有一组等待队列，表达复杂状态时容易让不同类型的等待线程混在一起。

`ReentrantLock` 可以创建多个 `Condition`，例如：

1. 一个条件专门等“队列非空”。
2. 另一个条件专门等“队列未满”。

这类生产者消费者模型里，多条件队列会让唤醒语义更清晰，也更容易减少无效唤醒。
如果你能把这个点讲出来，通常比泛泛而谈“更灵活”更有说服力。

## 公平锁可以提，但不要把它说成默认更好

`ReentrantLock` 支持公平锁，这也是常见面试点。但回答时最好顺手补一句：
公平锁强调先来先得，能降低线程长期饥饿的概率，却往往会牺牲吞吐。

所以公平不等于先进。大多数业务系统更关心整体吞吐和平均延迟，未必需要公平策略。
如果你只会说“`ReentrantLock` 支持公平锁，所以更强”，答案就显得偏教材式。

## 一种更完整的面试回答方式

如果让我在面试里简洁回答，我会这样组织：

`synchronized` 适合保护简单临界区，语法直接，自动释放锁，维护成本低；`ReentrantLock` 适合需要
超时尝试、可中断获取、公平策略或多个条件队列的场景。两者不是单纯性能高低之分，而是“更少控制、更低出错率”
和“更多控制、更多使用责任”之间的取舍。普通业务代码优先选简单方案，只有在同步语义确实不够时再上 `ReentrantLock`。

这样的回答，既有结论，也有判断依据，通常比单纯罗列 API 差异更像真实工程经验。
