## 先别急着把问题归因到“流量上来了”

面试里常见一种提问方式：某个列表查询原来几十毫秒，现在变成一两秒，你会怎么排查？
很多人的第一反应是“看 CPU、加机器、扩容数据库连接池”。这些动作并非永远错误，但如果一开始就往资源方向想，
往往会错过最关键的一层：这条 SQL 现在到底还在不在用合适的方式过滤数据。

查询变慢，本质上不是“机器不够努力”，而是数据库为了拿到结果，付出的扫描和回表成本突然变高了。
而这类成本变化，最常见的来源并不是用户量神秘地增长了一点，而是索引命中方式、过滤条件选择性、
排序分页策略或执行计划发生了变化。

## 为什么很多慢查询首先应该看索引过滤能力

数据库执行一条查询时，最重要的问题不是“有没有索引”，而是“这个索引能不能把无关数据尽早挡在外面”。
如果一个查询表面上也走了索引，但索引列选择性很差，或者过滤条件没法利用联合索引的前缀顺序，
最终依然可能扫出大量记录再回表过滤。此时从业务上看只是“同一个接口变慢了”，
从数据库角度看却是“为了取 20 条数据，多看了几万甚至几十万行”。

所以排查时，我通常先确认三件事：

1. 执行计划是否真的命中了预期索引。
2. 命中索引后扫描行数是否明显偏大。
3. where、order by、limit 的组合有没有破坏原本的高效过滤路径。

这一步之所以优先，是因为只要过滤能力出了问题，再多资源也只是帮系统更快地做无效扫描。

## 一个典型误区：以为“走了索引”就说明没问题

很多候选人在面试里会说“我先看 explain，如果 key 不为空就说明走索引了”。这回答不够。
`key` 不为空只能说明优化器选了某个索引，并不代表这条路径足够高效。

更值得看的是：

1. `rows` 预估扫描量是不是异常大。
2. `type` 是 `range`、`ref`、`const`，还是已经退化成 `index`、`ALL`。
3. `Extra` 里有没有 `Using filesort`、`Using temporary`、`Using where` 等提示。

如果原来几十行就能定位结果，现在要先扫描几万行，即便最后仍显示使用索引，性能也会明显下滑。
因此面试里最好把“是否走索引”升级成“索引是否仍在有效过滤和支撑排序”。

## 扩容能缓解压力，但通常不能修正错误的数据访问路径

为什么不建议第一步就扩容？因为扩容更多是在吞吐层面缓解资源争抢，比如 CPU 高、IO 饱和、连接数不足。
但如果一条 SQL 的访问路径本身已经退化，扩容只能把代价摊给更多资源，并不能让它重新变聪明。

举个简单例子，原来根据 `status`、`created_at` 的联合索引做分页查询，每次只扫几十行。
后来业务增加了一个函数处理，例如对时间列做了计算，或者把前导列条件改成范围查询，
优化器就可能无法继续高效利用后续列。结果是单次查询的扫描量暴涨。这个时候你加实例、加连接池，
可能只是把慢查询从“偶发超时”变成“暂时还能扛住”，但根因仍留在那儿。

## 面试里更稳的回答顺序应该是什么

如果我在面试中回答“查询变慢了怎么办”，我通常会按下面这条线讲：

先确认是否是单条 SQL 退化，而不是整体资源耗尽。随后用慢日志和 `EXPLAIN` 看执行计划，
重点关注是否命中预期索引、扫描行数有没有异常放大、排序和分页有没有走到临时表或文件排序。
如果发现过滤能力下降，就从索引设计、条件写法、联合索引顺序和回表成本入手优化。
只有当 SQL 访问路径本身合理、但整体负载仍超过当前资源上限时，才讨论扩容、读写分离或缓存。

这套顺序的好处是，你先在因果关系上站稳了：先修正数据访问路径，再判断资源是否真的不够。

## 业务变更经常比流量增长更容易让索引失效

线上很多慢查询并不是因为访问量突然翻倍，而是因为需求迭代悄悄改坏了原有查询形态。比如：

1. 增加了一个模糊匹配，导致无法继续使用原有前缀索引。
2. 在索引列外包了函数或表达式，让条件失去可索引性。
3. 新增筛选条件后，联合索引顺序不再匹配主要过滤路径。
4. 分页越来越深，`limit offset` 本身带来的跳过成本开始明显。

这些变化单看业务代码常常很小，但对数据库执行计划的影响可能非常大。面试官如果继续追问，
你能把这一层说出来，说明你理解的是“查询为何变慢”，而不是只记住了“慢了就扩容”。

## 最后用一句话收束

查询变慢时先看索引过滤，而不是先扩容，因为数据库性能问题首先取决于数据访问路径是否正确。
如果一条 SQL 已经在大量扫描无关数据，扩容只是提高做错事的资源上限；先修正索引与条件匹配关系，
通常才是更接近本质、更有工程收益的第一步。
