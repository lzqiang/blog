## 事件驱动写法为什么常在细节里出错

很多 Java 项目走到一定规模以后，都会把“下单成功后通知库存、营销、积分”这类动作拆成事件。
方向本身没有问题，问题往往出在事件发出的时机。业务代码刚把订单写入数据库，就顺手调用消息发送器，
看上去很自然，但只要事务后续回滚，外部系统就会收到一条本不该存在的事件。

这类问题在开发环境里不容易暴露。因为大多数时候数据库提交成功，消息也正好发出去了，
链路显得很顺滑。真正到线上，异常、超时、并发冲突一多，事务边界和事件边界不一致的问题就会开始冒头。

## 常见误区是把“方法执行成功”当成“事务已经成功”

在 Spring 里，只要方法还没离开事务提交阶段，就不能简单认为数据已经最终落库。
有些代码会在事务方法内部直接调用：

```java
orderRepository.save(order);
eventPublisher.publish(new OrderCreatedEvent(order.getId()));
```

如果 `publish` 只是进程内通知，风险还相对可控；但只要它继续把消息发到 MQ、Webhook 或其他服务，
你就等于把未最终确认的数据提前广播出去了。之后事务因为唯一键冲突、连接抖动或者人工抛出的业务异常而回滚，
外部世界却已经开始基于这条消息做下一步动作。

很多线上“数据对不上”的事故，根因都不是消息系统丢了消息，而是消息发得太早。

## Outbox 的核心不是组件，而是顺序

Outbox 模式经常被说得很重，好像必须引入一套复杂基础设施。其实它最关键的思想很朴素：
业务数据和“待发送事件”先在同一个本地事务里落库，等事务真正提交后，再由独立流程把事件投递出去。

这样一来，顺序就清楚了：

1. 订单写入成功。
2. 对应事件记录也写入 `outbox` 表。
3. 本地事务一起提交。
4. 后台投递程序扫描未发送事件并发送到消息系统。

只要第 3 步没有成功，第 4 步就不会发生。事务和事件边界终于对齐，不再靠运气。

## 设计 Outbox 表时，先把“补偿”想清楚

很多人第一次落地 Outbox，会只存一列事件内容 JSON，够用但不耐久。更稳妥的做法是至少保留这些字段：

- 事件 ID，用于幂等。
- 聚合根 ID，方便排查具体业务对象。
- 事件类型，区分订单创建、支付成功、库存释放等动作。
- 载荷内容，通常是 JSON。
- 发送状态，例如待发送、已发送、发送失败。
- 重试次数和最近错误信息，便于运维和告警。

这样设计的好处是，一旦下游消息系统短暂不可用，系统仍然可以把失败事件留在本地表中，稍后重试。
你不需要猜“刚才那条消息到底有没有发出去”，因为状态已经被显式记录下来了。

## 不要把 Outbox 当成“最终一致性万能药”

Outbox 解决的是“本地事务提交后如何可靠地把事件送出去”，但它不自动解决所有跨系统一致性问题。
比如下游消费者重复消费怎么办、多个事件是否必须保证顺序、某些事件是否需要人工补偿，这些都还是业务设计问题。

因此在 Java 项目里，Outbox 通常要和另外两件事配套：

1. 消费端幂等。即使同一事件被重复投递，业务结果也不能重复生效。
2. 可观测性。失败重试、积压数量、最长未发送时长都应该有监控。

如果只做了一张表，却没有重试和告警，Outbox 很容易沦为一块安静积灰的故障现场。

## 什么时候值得用，什么时候不必急着上

如果你的系统只是单体应用，事件也只在进程内做一些非关键通知，例如刷新本地缓存，
那不一定需要立刻上 Outbox。可一旦事件已经跨进程、跨服务，或者会驱动库存扣减、优惠券发放、
账务入账这类不可轻易撤回的动作，就应该认真对待事务边界。

判断是否需要 Outbox，我通常只看一个问题：数据库提交失败时，外部动作能不能容忍“已经发生”？
如果答案是否定的，那就别再让事件在事务里裸奔。

## 比起追求优雅，先保证边界正确

很多工程方案在白板上都很优雅，但真正决定系统稳不稳的，往往是边界处理是否诚实。
领域事件本身没错，错的是把它们发在了一个还没真正完成提交的时间点上。

先落库，再投递；先保证本地状态成立，再通知外部世界。这听起来像保守做法，
却往往是 Java 后端把复杂业务跑稳之后最值得坚持的工程纪律。
