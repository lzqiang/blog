## 别把“重复消费”只理解成消费者代码写得不严谨

面试里经常有人问：消息队列出现重复消费怎么办？不少回答会直接落到“做幂等”，
然后开始讲唯一键、去重表或者 Redis 标记。这些方向没错，但如果一上来就只说“消费者要幂等”，
通常还不够完整。

重复消费首先是一个现象，不是根因。你要先说明消息为什么会重复出现，
再决定应该把治理重点放在生产端、Broker 投递语义、消费确认逻辑，还是业务落库环节。
如果这一步没有讲清，后面的方案很容易显得只是背标准答案。

## 为什么要先分清重复发生在哪一段链路

一条消息从业务系统发出，到最终被消费成功，中间至少经过几段：

1. 生产端生成并发送消息。
2. Broker 接收、持久化并投递。
3. 消费端拉取或接收消息后执行业务逻辑。
4. 消费端提交确认，Broker 才把这条消息视为完成。

只要其中某一步出现超时、重试、网络闪断或状态不一致，重复就可能发生。
比如生产者发送成功但没有收到确认，可能会重发；消费者已经处理完业务，
但确认提交失败，Broker 也可能再次投递。这里的关键点在于：
消息系统大多只能尽量保证“至少一次”，重复本身不是异常，而是语义的一部分。

所以面试里更稳的表达应该是：我会先判断重复发生在发送侧、投递侧还是消费确认侧，
因为不同阶段对应的治理重点并不一样。

## 真正让面试官买账的，是你知道幂等要落在哪里

很多人会说“加一个幂等表就好了”，但如果没有业务上下文，这句话其实很空。
幂等并不是把所有消息都查一遍表，而是要结合业务动作来设计判断边界。

例如订单支付结果通知、库存扣减、优惠券发放，这三类消息都可能重复，
但它们的幂等落点不一样：

1. 支付通知更适合围绕业务单号和状态流转做防重，避免重复推进订单状态。
2. 库存扣减通常要结合唯一请求号、事务和扣减条件，防止多次减少库存。
3. 优惠券发放则往往依赖用户、活动、券模板等组合约束，避免重复发券。

也就是说，幂等不是一层通用胶水，而是业务状态机的一部分。
你在面试里如果能把这一点说出来，回答会明显比“消费者里先查 Redis”更扎实。

## 一个常见误区：只做消费前检查，却不保证检查和落库原子

很多初学者会这样设计：

1. 收到消息。
2. 先查某个去重表或 Redis Key 是否存在。
3. 不存在就执行业务。
4. 执行完再写入去重标记。

这种写法在单机场景里看似够用，实战里却有明显窗口。
如果两个消费者并发处理同一条重复消息，可能同时查到“还没处理过”，
最后双双进入业务逻辑。问题不在于你有没有“查重”，而在于查重和业务提交是不是一个原子动作。

所以更稳的做法通常是把幂等约束下沉到数据库唯一索引、状态迁移条件更新，
或者与业务事务放在同一个提交边界内。这样即使消息真的来了两次，
系统也会在最终写入阶段挡住第二次副作用。

## 面试里可以顺手讲清：为什么不能一味追求“绝不重复”

如果对方继续追问，你还可以补一句：消息系统在工程上通常更容易做到至少一次，
而不是绝对不重复。为了追求“绝不重复”，往往要引入更高的协调成本、
更复杂的状态管理，甚至牺牲可用性和吞吐。

这时候你的重点不应该是承诺“重复永远不会发生”，而是说明：

1. 链路设计上尽量减少不必要的重复发送和重复投递。
2. 业务侧接受重复可能发生，并把副作用控制在一次。
3. 监控上能识别重复率是否异常升高，避免上游故障长期放大。

这样的回答会让面试官看到你理解的是分布式系统取舍，而不是只会背一个术语。

## 如果要给出一套完整回答，可以按这个顺序组织

先说明消息重复消费是消息系统至少一次语义下的常见现象，不一定意味着系统出错；
然后区分重复来源，是生产端重发、Broker 重投还是消费确认失败；
接着说明业务侧必须设计幂等，而且幂等最好绑定业务唯一标识、状态流转和事务边界，
不能只靠消费前查一次缓存；最后再补充监控和补偿，比如统计重复率、
异常重试次数和死信情况，防止重复从个别现象变成系统性噪声。

## 最后用一句话收住

面试被问消息重复消费时，先分清重复来源再谈幂等，比直接说“做个去重表”更有说服力。
因为真正要解决的不是“消息为什么又来了一次”这件事本身，
而是系统在它又来一次时，能不能依旧只产生一次正确的业务结果。
