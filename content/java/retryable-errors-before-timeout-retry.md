## 许多线上抖动，不是因为没有重试，而是因为重试太早

Java 服务治理里，超时和重试几乎总是成对出现。调用下游接口偶尔慢了，大家第一反应往往是把超时时间调短一点，
再补一层自动重试，觉得这样能把偶发抖动兜住。这个思路并不完全错，但它有个常见前提没有被说出来：
只有当失败属于“短暂且可恢复”的类型时，重试才有价值。

如果根因并不是瞬时故障，而是参数错误、库存不足、幂等约束冲突或者下游已经开始雪崩，
那你多发一次请求，得到的通常不是恢复，而是把压力再放大一轮。很多系统不是第一次失败时垮掉的，
而是在自动重试把原本局部的问题扩散以后，才真的进入不可控状态。

## 先把异常分成三类，很多决策会清楚很多

设计重试策略时，我通常先把异常粗分成三类。

第一类是瞬时故障。比如网络抖动、连接池里个别连接失效、下游实例短暂切换、偶发超时但整体成功率正常。
这类问题存在“下一次可能成功”的条件，适合有限次重试。

第二类是明确业务拒绝。比如校验失败、余额不足、状态不允许重复提交、权限不足。
这类异常通常不是靠时间能恢复的，重试只会浪费资源，还会让调用方误以为系统在努力补救。

第三类是结果未知的副作用异常。最典型的是请求在客户端超时了，但下游可能已经执行成功，只是响应没回来。
这时候如果直接重试，最危险的问题不是“又失败一次”，而是把扣款、发券、发货之类动作做了两遍。

真正稳健的重试策略，不是统一 catch 一切异常后再来一次，而是先判断失败属于哪一类。

## Java 里最容易踩的坑，是把技术异常和业务异常混在一起

很多项目的 service 层会直接抛出各种运行时异常，调用方再统一在切面或框架层做重试。
表面上看省事，实际上把语义抹平了。`SocketTimeoutException` 和“订单状态不允许取消”都可能最后包成一个
`RuntimeException`，框架层根本分不清哪一种值得重试。

如果团队已经使用 Spring Retry、Resilience4j 或网关侧重试，更要注意这个边界。框架可以负责“如何重试”，
但“什么能重试”最好由业务代码或适配层先表达清楚。否则默认策略很容易变成：
只要抛异常就重试，只要超时就重试。这个策略在演示环境里看起来成功率更高，在真实高峰期却经常成为放大器。

更实际的做法是为外部调用建立明确的异常模型，例如：

1. 可重试的连接类异常和短暂超时。
2. 不可重试的参数类、鉴权类、业务状态类异常。
3. 结果未知、必须依赖幂等键或查询确认的异常。

只要这层语义先分清，后面的超时阈值、重试次数和退避策略才有讨论基础。

## 超时之后是否重试，关键看接口是不是幂等

很多人讨论重试时只盯着失败原因，却忽略了另一个更关键的问题：这个接口重试以后，副作用能不能被安全吸收。
比如查询类接口通常天然适合重试；而创建订单、扣库存、发送短信这类写操作，如果没有幂等保障，
哪怕只重试一次，也可能带来重复执行。

所以在 Java 服务之间约定超时重试前，我更倾向先问两个问题：

1. 调用失败时，是否存在下一次立刻成功的可能。
2. 即使重复触发，下游是否能通过业务唯一键、请求号或状态机保证结果不重复生效。

如果第二个问题答不上来，重试策略就不该先落到代码里，而应该先补幂等设计。否则所谓容错，
只是把风险从“偶发失败”换成“低概率重复副作用”。

## 重试次数越少越好，退避比硬顶更重要

很多线上事故都说明了一个朴素事实：当下游已经吃紧时，最危险的不是失败本身，而是上游同时决定“再试一次”。
几十个线程瞬间把一次调用变成三次调用，流量就会被主动放大成原来的几倍。

因此重试策略应当尽量保守：

1. 次数要少，通常一到两次已经是上限。
2. 间隔要有退避，而不是立即重发。
3. 总超时时间要受控，别让一次用户请求在内部拖出更长链路。
4. 触发重试时要打点，能看见具体接口、异常类型和重试后结果。

这里真正的目标不是“把所有失败都藏起来”，而是在不显著放大系统压力的前提下，吸收少量瞬时故障。

## 比起先写注解，先把失败语义讲明白

在 Java 项目里，给方法补一个重试注解通常只要几分钟，但决定哪些异常值得重试、哪些超时只能走补偿，
往往需要团队认真讨论调用语义。后者更慢，却也更重要。

超时重试之前，先分清异常是否真的可恢复、接口是否具备幂等、副作用是否能够确认。
只有这三件事站稳了，重试才是系统弹性的一部分；否则它更像一把没区分场景就随手按下去的放大器。
