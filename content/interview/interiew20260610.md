# 2026-06-10面试题

## 1. `HashMap` 和 `TreeMap` 有什么区别？

**答案：**

`HashMap` 基于哈希表，平均查询复杂度接近 O(1)，不保证遍历顺序，允许一个 `null` key。

`TreeMap` 基于红黑树，查询复杂度为 O(log n)，会按照 key 的自然顺序或指定的 `Comparator` 排序，通常不允许 `null` key。

选择建议：

- 需要快速按 key 查询时使用 `HashMap`。
- 需要排序、范围查询或查找邻近 key 时使用 `TreeMap`。

既然 `TreeMap` 使用红黑树，下一题继续看：红黑树为什么适合工程实现？

## 2. 红黑树为什么比普通二叉搜索树稳定？

**答案：**

普通二叉搜索树在连续插入有序数据时，可能退化成链表，查询复杂度从 O(log n) 降为 O(n)。

红黑树通过颜色规则和旋转操作保持近似平衡，使树高始终受控，因此查询、插入和删除的复杂度都能保持在 O(log n)。

红黑树没有追求绝对平衡，所以插入、删除时的调整成本通常低于严格平衡树。

它适合需要频繁增删查的通用场景，例如：

- `TreeMap`
- `TreeSet`
- JDK 8 中发生严重哈希冲突后的 `HashMap`

树结构解决了查找效率问题，而并发读写还需要安全控制。下一题看读写锁。

## 3. `ReentrantReadWriteLock` 适合什么场景？

**答案：**

`ReentrantReadWriteLock` 提供读锁和写锁：

- 多个线程可以同时持有读锁。
- 写锁与其他写锁互斥。
- 写锁与所有读锁互斥。

它适合读多写少，并且临界区执行时间相对较长的场景。

如果写操作频繁或临界区很短，维护读写锁的成本可能高于普通互斥锁。

还要注意锁升级问题。线程持有读锁时直接申请写锁，可能导致死锁。通常应先释放读锁，再尝试获取写锁。

读多写少场景还可以使用乐观读。下一题继续看 `StampedLock`。

## 4. `StampedLock` 和读写锁有什么区别？

**答案：**

`StampedLock` 支持三种模式：

- 写锁。
- 悲观读锁。
- 乐观读。

乐观读不会真正阻塞写线程。读取数据后，需要使用 stamp 校验读取期间是否发生过写入；如果校验失败，再升级为悲观读。

```java
long stamp = lock.tryOptimisticRead();
int value = data;

if (!lock.validate(stamp)) {
    stamp = lock.readLock();
    try {
        value = data;
    } finally {
        lock.unlockRead(stamp);
    }
}
```

`StampedLock` 的特点：

- 乐观读可以提高读多写少场景的吞吐量。
- 它不可重入。
- 使用方式比普通读写锁复杂。
- 必须正确保存和释放 stamp。

并发控制不仅涉及锁，还涉及无锁容器。下一题看 `ConcurrentLinkedQueue`。

## 5. `ConcurrentLinkedQueue` 如何实现线程安全？

**答案：**

`ConcurrentLinkedQueue` 是基于链表实现的无界非阻塞队列。

它主要通过 CAS 更新头尾节点，不需要给整个队列加互斥锁。竞争失败的线程会重新读取状态并重试。

优点：

- 无锁实现，高并发下通常具有较好吞吐量。
- 不会因为一个线程持有锁而阻塞所有其他线程。

限制：

- 它是无界队列，无法通过容量形成反压。
- 不提供阻塞等待能力。
- `size()` 需要遍历队列，开销较大。

需要阻塞或明确容量限制时，可以选择 `ArrayBlockingQueue`、`LinkedBlockingQueue` 等。

## 6. `ArrayBlockingQueue` 和 `LinkedBlockingQueue` 如何选择？

**答案：**

`ArrayBlockingQueue` 基于固定长度数组，创建时必须指定容量。

特点：

- 容量边界明确。
- 内存连续。
- 不需要为每个元素创建链表节点。
- 生产和消费通常竞争同一把锁。

`LinkedBlockingQueue` 基于链表，可以指定容量。

特点：

- 生产和消费使用不同锁，并发吞吐可能更好。
- 每个节点有额外内存开销。
- 如果不指定容量，默认容量非常大，容易导致任务堆积和 OOM。

选择建议：

- 强调严格容量和可控内存时，优先考虑 `ArrayBlockingQueue`。
- 生产和消费并发较高、容量也有明确限制时，可以考虑 `LinkedBlockingQueue`。
- 生产环境必须明确队列容量。

队列容量会直接影响线程池行为。下一题看线程池队列容量如何设计。

## 7. 线程池队列容量应该如何设计？

**答案：**

线程池队列容量应结合以下因素设计：

- 峰值任务到达速率。
- 单个任务平均处理时间。
- 允许的最大等待时间。
- 下游服务承载能力。
- 最大可用内存。

可以用 Little's Law 做初步估算：

```text
系统中的平均任务数 ≈ 任务到达速率 × 平均停留时间
```

但最终仍需要通过压测验证。

队列太小会导致频繁拒绝；队列太大则会隐藏系统过载，让任务长时间排队，最终拉高 P99 延迟甚至导致 OOM。

需要同时监控：

- 当前队列长度。
- 任务等待时间。
- 活跃线程数。
- 拒绝次数。
- 任务执行时间。

队列积压通常首先表现为尾延迟升高。下一题看平均耗时正常但 P99 很高的原因。

## 8. Java 接口平均耗时正常，但 P99 很高，可能是什么原因？

**答案：**

平均值会掩盖少量非常慢的请求。常见原因包括：

- 偶发 Full GC。
- 数据库连接池等待。
- 线程池任务排队。
- 数据库锁等待。
- 少量慢 SQL。
- 下游接口超时和重试。
- 热点缓存失效。
- 少量超大请求或返回数据。

排查时应该抽取 P99 请求的 traceId，分解以下耗时：

- 线程池等待时间。
- 数据库执行时间。
- Redis 访问时间。
- HTTP 下游调用时间。
- JSON 序列化时间。

再把慢请求的时间点与 GC、CPU、连接池和线程池指标对齐。

如果慢请求集中发生在缓存失效时，下一题就需要考虑热点 key 重建问题。

## 9. 热点缓存失效时如何避免大量请求回源？

**答案：**

可以使用互斥重建，只允许一个线程查询数据库并刷新缓存，其他线程等待、快速失败或暂时返回旧值。

还可以使用逻辑过期：

1. 缓存中保存业务数据和逻辑过期时间。
2. 数据逻辑过期后，前台仍返回旧数据。
3. 后台线程异步刷新缓存。

对于极少变化的热点数据，也可以不设置物理过期时间，通过定时任务主动刷新。

无论使用哪种方式，都要考虑：

- 重建锁的超时时间。
- 获取锁失败后的行为。
- 数据库查询失败的兜底。
- 后台刷新任务异常。
- 热点 key 识别和监控。

缓存保护失败后，大量请求可能击穿数据库。最后一题看如何快速止损。

## 10. 热点缓存击穿导致数据库负载飙升，如何应急处理？

**答案：**

首先要限流或降级，阻止新增请求继续压垮数据库。

然后恢复热点缓存：

- 人工预热热点 key。
- 启动单线程或受控并发重建。
- 临时返回旧数据或默认数据。

同时要关闭不必要的重试，避免流量被进一步放大。

接下来检查：

- 数据库连接池是否耗尽。
- 是否出现慢 SQL。
- 是否存在锁等待。
- 非核心定时任务是否占用数据库。
- 下游请求是否持续重试。

必要时暂停非核心任务，优先保护核心业务。

故障恢复后，应补充：

- 热点 key 识别。
- 互斥重建。
- 逻辑过期。
- 缓存预热。
- 过期时间随机抖动。
- 数据库限流和熔断。
- 故障压测与演练。
